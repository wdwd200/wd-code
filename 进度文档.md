# 进度文档

当前项目是一个最小命令行编程助手 demo。它保持 `wdcode` 启动方式不变，但源码已经统一放入 `src/` 目录，并把配置处理单独拆成 `src/config.py`。

## src/cli_assistant.py

`parse_args(argv)` 负责定义和解析命令行参数。它只处理入口层需要的参数，包括模型地址、密钥、模型名、配置文件路径和是否保存配置。

`main(argv=None)` 是程序主入口。它先解析命令行参数，再调用配置层生成运行配置，然后创建 `LLMClient` 和默认工具注册表，最后启动 `run_agent_loop` 进入命令行对话。

## src/config.py

`load_config(path)` 负责读取 JSON 配置文件。配置文件不存在时返回空配置；配置文件格式错误或内容不是 JSON 对象时，会终止程序并显示错误。

`save_config(path, config)` 负责保存配置。它会确保目标目录存在，并把最终配置写成 JSON 文件。

`first_value(*values)` 负责从多个候选配置值中选出第一个有效值，用来维持当前配置优先级。

`normalize_base_url(base_url)` 负责清理模型服务地址末尾多余的斜杠。

`build_runtime_config(args)` 负责合并运行时配置。配置优先级保持不变：命令行参数优先，其次是环境变量，再其次是 `model_config.json`，最后只对 `base_url` 使用默认值。它还会检查必填字段是否完整，并在需要时保存配置。

## src/agent_loop.py

`run_agent_loop(client, tool_registry=None)` 负责主循环。它读取用户输入，调用命令层判断是否需要退出，通过对话状态层维护 `messages`，调用 `client.chat(messages, tools=...)` 获取模型回复，并把最终回复输出到命令行。传入工具注册表后，它会把工具 schema 交给模型。

`handle_assistant_message(client, conversation, assistant_message, tool_registry)` 负责处理模型返回的 assistant message。如果没有 `tool_calls`，它会把普通文本回复写入对话历史并返回；如果有 `tool_calls`，它会执行工具，把工具结果写回对话历史，然后再次调用模型生成最终回复。

`execute_tool_call(tool_registry, tool_call)` 负责执行单个 OpenAI 风格工具调用。它解析工具参数 JSON，通过 `ToolRegistry` 调用具体工具，并把工具执行结果返回给上层。如果具体工具执行失败，会把错误信息包装成工具结果。

## src/conversation.py

`Conversation.__init__(system_prompt=SYSTEM_PROMPT)` 负责创建本次运行期间的对话状态。它会先放入一条 system 消息。这里的状态只存在于内存中，不做长期保存。

`Conversation.add_user_message(content)` 负责把用户输入追加成一条 user 消息。

`Conversation.add_assistant_message(content)` 负责把模型回复追加成一条 assistant 消息。

`Conversation.add_assistant_tool_call_message(message)` 负责把包含 `tool_calls` 的 assistant message 原样加入消息历史，保持 OpenAI tool calling 所需的上下文格式。

`Conversation.add_tool_result(tool_call_id, name, content)` 负责把本地工具执行结果写回消息历史。它会生成 role 为 `tool` 的消息，并关联模型请求里的 `tool_call_id`。

`Conversation.remove_last_message()` 负责删除最后一条消息。当前用于模型请求失败时撤回刚刚追加的用户消息，避免失败请求影响后续对话。

`Conversation.as_messages()` 负责把内部消息列表交给模型客户端使用。

## src/commands.py

`is_exit_command(user_input)` 负责识别本地退出命令。当前只识别 `/exit` 和 `/quit`。它不调用外部工具，也不处理模型请求。

## src/llm_client.py

`LLMClient.__init__(base_url, api_key, model, timeout=120)` 负责保存模型请求所需的基础参数。

`LLMClient.chat(messages, tools=None)` 负责调用 OpenAI 兼容的 `/chat/completions` 接口。它组装请求、发送 HTTP 请求、处理网络或接口错误，并返回完整 assistant message。传入 `tools` 时，它会把工具 schema 一起发给模型。

## src/tools/__init__.py

`create_default_registry(project_root)` 负责创建默认工具注册表。当前它只注册只读文件工具，不包含写文件、执行命令或应用补丁能力。

## src/tools/tool.py

`Tool` 是工具结构定义。它保存工具名称、说明、参数 schema 和执行函数。

`Tool.to_openai_schema()` 负责把本地工具结构转换成 OpenAI tool calling 兼容的 schema 形状，方便后续接入模型工具调用。

## src/tools/registry.py

`ToolRegistry.__init__()` 负责创建空的工具注册表。

`ToolRegistry.register(tool)` 负责注册工具。如果同名工具已经存在，会直接报错，避免覆盖已有工具。

`ToolRegistry.get(name)` 负责按名称查找工具。找不到时返回空值。

`ToolRegistry.list_tools()` 负责返回当前已经注册的工具列表。

`ToolRegistry.schemas()` 负责返回所有工具的 OpenAI 兼容 schema。

`ToolRegistry.execute(name, arguments)` 负责执行指定工具。它只根据名称找到工具并调用工具自己的执行函数，不直接实现具体工具能力。

## src/tools/files.py

`create_file_tools(project_root)` 负责创建第一批只读文件工具。当前包含 `list_files` 和 `read_file`。

`resolve_project_path(project_root, user_path)` 负责把用户传入的项目相对路径解析成真实路径，并确保路径仍然位于项目根目录内，避免越界访问。

`list_files(project_root, arguments)` 负责列出指定项目相对目录下的文件和子目录。它只返回名称、相对路径和类型，不读取文件内容。

`read_file(project_root, arguments)` 负责读取指定项目相对文本文件的内容。它只允许读取常见文本扩展名文件，不提供写入能力。

## 启动文件

`wdcode.cmd` 和 `wdcode.bat` 仍然是 Windows 启动入口。它们现在会调用 `src/cli_assistant.py`，所以用户在项目目录的 cmd 里仍然可以直接输入 `wdcode` 启动项目。

`model_config.json` 仍然放在项目根目录，默认由 `src/config.py` 读取。

## 当前能力边界

当前完成了源码目录整理、配置层拆分、对话状态层拆分、本地命令层拆分，以及只读工具层的最小闭环。模型现在可以收到 `list_files` 和 `read_file` 的工具 schema；当模型返回 OpenAI 风格 `tool_calls` 时，agent loop 会执行对应只读工具，把结果写回消息历史，并再次调用模型生成最终回复。项目暂时不实现写文件、命令执行、补丁应用、自动修改代码、长期记忆、复杂任务规划或多 agent 协作。
